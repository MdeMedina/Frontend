# Impeccable Design Context

## Design Context

### Users
This is a **residential management platform** used by four distinct roles:

- **ADMIN**: Property administrators focused on operational control. Primary job: reviewing and approving/rejecting petitions. Need fast, actionable data at a glance.
- **OWNER (Propietario)**: Property owners who primarily manage reservations and oversee their units. Need a clean, high-visibility calendar interface without distraction.
- **CONCIERGE (Conserje)**: Front-desk staff managing floor control in real time. Need a high-contrast status grid for rapid visual scanning.
- **ASSIGNED_MANAGER (Responsable)**: Delegates of owners managing unit-level operations.
- **SUPERADMIN**: Platform-level governance, managing residences and administrators.

**Context of use**: Desktop-first, professional workspace environment. Users are repetitively managing data throughout their workday. They need clarity and speed, not entertainment.

### Brand Personality

**Three words**: Confianza · Precisión · Rigor Técnico

**Emotional goal**: The interface should evoke the same confidence you feel using a surgical instrument—every element is exactly where it needs to be, nothing more, nothing less. Users should feel **in control and competent**, never overwhelmed.

**Anti-references**: Avoid anything that feels casual, playful, or startup-friendly (no gradients, no rounded blobs, no emoji, no decorative illustrations).

### Aesthetic Direction

**Visual tone**: "Surgical and High Precision" — clinical whitespace, defined edges, structured hierarchy.

**Theme**: Light mode only. Pure white surfaces on a cold gray background.

**References**: Think Bloomberg Terminal UI discipline meets modern SaaS precision (Linear, Vercel dashboard).

#### Color Palette
| Token | Value | Usage |
|---|---|---|
| `primary` | `#002D72` | Primary actions, active states, brand identity |
| `primary-subtle` | `rgba(0, 45, 114, 0.1)` | Active navigation backgrounds |
| `success` / `action` | `#A8E6CF` | Confirmation, success states, action affordance |
| `surface` | `#FFFFFF` | Cards, modals, panels |
| `background` | `#F8FAFC` | Page background (cold surgical gray) |

#### Border Radius
- **Rule**: Maximum `rounded-sm` (2px–4px) everywhere. No `rounded-md`, `rounded-lg`, or `rounded-full` on structural elements.
- **Rationale**: Defined angles communicate technical rigor. Soft radii signal casualness.

#### Spacing & Density
- **Low Information Density**: Maximum 5 focal elements per view.
- **Minimum padding**: `p-8` or `p-10` on all primary containers.
- **Generous whitespace**: Breathing room is a feature, not waste.

### Design Principles

1. **Clarity over density**: Show only what is needed to make the next decision. Remove anything decorative.
2. **Actions live with objects**: Buttons (Aprobar, Reservar, Cancelar) must emerge directly from their parent Cards — never hidden in dropdowns or secondary menus.
3. **Icons, never emoji**: All navigation icons must use Lucide React (already installed) with `strokeWidth={1.5}`. No Unicode emoji in UI chrome.
4. **Tailwind governs all**: Zero inline `style={{...}}` attributes. Every visual decision lives in `tailwind.config.js` and utility classes.
5. **Scale signals priority**: The primary CTA and the critical input field (e.g., Password on Login) must be visually scaled at 1.25x relative to secondary elements.

## Final Polish Rules

These rules are **mandatory constraints** applied as the last quality gate before shipping any UI change.

### 1. Zero Shadow Policy
- **Strictly forbidden**: `shadow-sm`, `shadow-md`, `shadow-lg`, `drop-shadow`, or any diffuse box-shadow.
- **Replacement**: All depth and separation must be expressed through `border border-slate-200` (1px solid border).
- **Rationale**: Surgical precision means hard edges, not soft halos.

### 2. Surgical Feedback (Icon Enforcement)
- Every icon in the application must come from `lucide-react` with `strokeWidth={1.5}`.
- **All emoji replacements in `Layout.tsx`** are required:
  - `'🏠'` → `<Home strokeWidth={1.5} />`
  - `'👥'` → `<Users strokeWidth={1.5} />`
  - `'🏗️'` → `<Building2 strokeWidth={1.5} />`
  - `'🏢'` → `<Building strokeWidth={1.5} />`
  - `'📅'` → `<CalendarDays strokeWidth={1.5} />`
  - `'📝'` → `<ClipboardList strokeWidth={1.5} />`
  - `'🗓️'` → `<Calendar strokeWidth={1.5} />`
  - `'📋'` → `<FileText strokeWidth={1.5} />`
  - `'☰'` (mobile menu) → `<Menu strokeWidth={1.5} />`

### 3. CSS Purge
- **Zero inline styles**: No `style={{...}}` in any JSX file. `Login.tsx` is the primary offender.
- **No `index.css` animations**: Any keyframe animation or transition defined in `index.css` must be migrated to Tailwind's `animate-*` utilities or defined in `tailwind.config.js` under `theme.extend.keyframes`.
- **token-only values**: No hardcoded hex colors in JSX/TSX. Use only design tokens defined in `tailwind.config.js`.

### 4. Mobile-First Thumb Reach
- The primary action for each role must be anchored for thumb reach on mobile:
  - **ADMIN**: A fixed bottom-right FAB (Floating Action Button) linking to `petitions`.
  - **OWNER**: A fixed bottom-right FAB linking to `reservations`.
- Use `fixed bottom-6 right-6 z-50` positioning.
- The FAB must use the `primary` (#002D72) color and a Lucide icon. No emoji.

## Standing Instructions

When generating **any** new component or refactoring `Layout.tsx`, `Login.tsx`, or `App.tsx`:

- **Cleanliness over function density** — if unsure between adding a feature or keeping it clean, keep it clean.
- Use only Lucide React icons with `strokeWidth={1.5}`.
- Apply design tokens from `tailwind.config.js`, never hardcoded hex values in JSX.
- Respect `rounded-sm` as the maximum border radius for structural elements.
- Contextual actions (Approve, Reserve, Cancel) must be embedded in cards, not in hidden menus.
- Apply all **Final Polish Rules** above before considering any component "done".
